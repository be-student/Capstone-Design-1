## Top 3 issues remaining

### 1. **P0 BLOCKER — Page 13 crashes on `KeyError: 'priority_rank'` (cascades to 13a expander + entire 13b + entire 13c)**

**Root cause:** schema mismatch between G1's pipeline artifact and the dashboard view.
- G1 wrote `results/retention_offers.csv` with columns `customer_id, segment, risk_level, churn_probability, recommended_action, offer_type, offer_detail, expected_uplift, estimated_cost_krw, expected_revenue_saved_krw, priority_score` (verified `head -1`).
- `src/dashboard/data_loader.py:1734` declares the empty-frame schema as `priority_rank` (a column that G1 never emits).
- `src/dashboard/app.py:4775` calls `filtered.sort_values("priority_rank")` and 4919 selects `priority_rank` for display.
- Tabs in `st.tabs(...)` execute sequentially in one script run, so the exception in tab_b also blanks tab_c (PNG `13_realtime_scoring_c_monitoring.png` shows only the 13b traceback).

**Fix:** rename one side. Either (a) update `data_loader.py` + `app.py` to use `priority_score`, or (b) have G1 emit `priority_rank` (an integer rank derived from `priority_score` desc).

### 2. **P0 BLOCKER — Page 09 Cost-Benefit shows ₩0 / 0.00x ROI because of a second schema mismatch**

- G1 emits `expected_revenue_saved_krw` (past participle "saved").
- `recommendations_view.py:579,611,669,698,704,765` and `app.py:4814,4868,4900,4922` reference `estimated_revenue_save_krw` (infinitive "save", legacy naming).
- The views guard with `if "estimated_revenue_save_krw" in offers.columns`, so they don't crash — they silently substitute `0.0`, leading to the ₩0 / ₩0 / 0.00x KPI tiles visible in `09_recommendations.png`.
- iter13 G4 fix log explicitly mentions "Total Campaign Cost ₩1.21M, Est. Revenue Saved ₩10.89M, ROI 9.0x" — that was the pre-fix synthetic value. The fix gated the synthetic away but did not align the column name, so post-fix the page renders zeros.

**Fix:** rename to a single canonical column. Same family as issue #1 — picking one name and updating both the producer (`src/models/recommendations.py::to_retention_offers`) and the consumers (`data_loader.py`, `app.py`, `recommendations_view.py`) closes both blockers.

### 3. **P1 — Drift trend charts degenerate to a single timestamp on Pages 08, 13c, 15 because `run_monitor` was executed once**

- All 34 rows in `results/drift_history.csv` carry timestamp `2026-05-11T16:39:29.329603+00:00` (verified via `awk` on the CSV).
- `drift_trend_guard` is documented as requiring "≥ 5 observations spanning ≥ 1 hour", but the actual chart in `08_model_monitoring.png` rendered anyway — it just shows dashed threshold lines and a single data marker, which is visually indistinguishable from broken-and-static.
- The 34-rows-real claim from pytest is technically true (count + columns + alert-level distribution are real), but the **time-series view of drift** is not yet meaningful.

**Fix:** either tighten `drift_trend_guard` to check the timestamp range (not just the count) and show the "insufficient history" banner instead of the empty chart, OR append a second `run_monitor` to populate two timestamps so the trend line actually has slope. (G1's smoke test mentioned a second monitor run produced 68 rows — that data was not the one shipped to the PNG.)
